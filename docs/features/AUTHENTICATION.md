# Authentication and Session Management

This document defines the architecture of the `auth` feature located in `src/features/auth`. Auth is a critical component implemented across the Next.js Middleware, Server Actions, and React Context.

## Authentication Strategy

Watch Rudra implements a credential-based architecture mapped over secure HTTP-only cookies.

### Dual-Factor Sign-In
1. **Password-Based Logic:** Standard login credentials stored using bcrypt.
2. **One-Time Password (OTP):** An email-based fallback or primary factor. Generated by the Node.js backend using `nodemailer`.

The `AuthForm` component (`src/features/auth/components/AuthForm.tsx`) dynamically morphs between asking for a generic password vs. displaying a 6-digit OTP input overlay.

### Next.js Middleware (The Gatekeeper)
Every client navigation is intercepted by `middleware.ts` located at the root of the project.
- **Protected Routes:** E.g., `/profile`, `/watch-party/[id]` immediately intercept unauthorized requests and redirect them to `/login` with an injected `?callbackUrl` query parameter.
- **Session Verification:** Middleware reads the encrypted session `token` directly out of the `cookies()` API. Note: Due to edge runtime restrictions, deep token validation happens on the Node.js backend.

## Anti-Bot Protection

To protect the OTP mailer and credential endpoints from credential-stuffing (brute-force attacks):
- **Cloudflare Turnstile:** Acts as an invisible CAPTCHA. Embedded directly in the AuthForm, no signup/login payload is accepted on the backend unless `turnstileToken` is parsed and validated by Cloudflare's servers.
- Locally simulated during testing where Turnstile logic is bypassed entirely via `.env.test`.

## The `useAuth` Provider
In `src/providers/auth-provider.tsx`, the global `AuthContext` injects the authorized User object context down the application tree. It implements an optimistic load state. If `loading` is true, the UI usually suspends or returns generic skeletons.
